College of Business App (code on github)

Introduction:

Currently, OSU's process of disseminating information and signing up for events is inconvenient for many students and faculty members. Because of the need for a smoother process in this area, the business school worked with our team to create an application focused on school activities. This work will use the React Native framework to build applications that simplify the current release activity flow and student check-in process. The app is planned to be built for IOS, Android, and Web.

Project vision:

1. Growth Hypothesis: Students and Staff at Oregon State will break the communication barrier that is currently in place surrounding events.

2. Value Hypothesis: Attendance at Oregon State events will increase, creating possibilities for more events to take place.

3. Functional requirements: The project can offer different functions for students and administrators.

4. Non-functional requirements: Students can check in by swiping a student ID card or scanning QR-code.

Tools and Technical Approaches:

1. React Native：

React Native is the fastest choice for developing applications for the web, Android and iOS. React Native allows developers to view their code and implementation on a real mobile screen. Our team members like React Native because it uses JavaScript language and styling similar to CSS to develop mobile apps, something each member was already familiar with. This allows us to develop quickly without having to learn tools for each platform. Another advantage of React Native is it’s modular design, allowing us to use free 3rd party tools.

2. Heroku：

This was the best choice for setting up our Express server because it was free to use, easy to set up, and could scale if we needed it. The only downside to using the free version is the servers need time to boot up before loading the content, so there is a 10 second wait if the server hasn't been used recently.

3. Postgres:

The advantage of Postgres was it is high documentation and Heroku's integration with it. It was a free to use database that had many easy to use 3rd party GUIs to interact with it.

4. AdobeXD:

This was the best choice we could have made for planning/designing the application. It's ability to create an interactable interface helped everyone involved to not only see the user interface but also the user experience of the application.
